## Context

`shiftOptimizer.ts` の評価関数（`evaluate()`）は辞書式比較（`isBetter()`）を用いて解の良し悪しを判定する。現在の評価基準は「不足ピーク → 不足合計 → ヘルプスタッフ日数 → 公平性 → 駐車場ピーク」の順で、アサイン枠を超えた場合のペナルティが存在しない。

この結果、最適化エンジン（Hill Climbing）はスタッフを多く出勤させる解を「良い解」と判断し続け、必要人数を超えるアサインが放置される。「全時間帯出勤の原則」（出勤すれば全スロットにアサイン）と組み合わさると、不要なスタッフが出勤することで複数スロットで同時に超過が発生する。

## Goals / Non-Goals

**Goals:**
- アサイン枠を超えるシフトを最適化によって抑制する
- 辞書式比較に `excessTotal` を追加し、超過のない解を超過のある解より優先させる
- 不足解消（shortfall）を超過抑制（excess）より高優先に維持する

**Non-Goals:**
- グリーディアルゴリズム（`autoShiftGenerator.ts`）の変更（「全時間帯出勤の原則」を変更しない）
- 超過を完全にゼロにすることの保証（不足解消を優先するため、不足解消のために超過が生じる場合は許容）

## Decisions

### 決定1: `excessTotal` を `shortfallTotal` の直後に挿入する

**採用**: `shortfallPeak → shortfallTotal → excessTotal → helpStaffTotal → fairnessVariance → parkingPeak`

**理由**: 不足（Shortfall）は人手不足であり超過（Excess）より業務上深刻なため、不足基準を優先する。ただし不足が同値の場合は超過を次に解消すべき問題として扱う。ヘルプスタッフの最小化・公平性・駐車場ペナルティより優先することで、「余計に出勤させない」原則を担保する。

**却下した代替案**:
- `excessTotal` を評価基準の最後尾に追加 → 公平性・駐車場より優先度が低くなり、超過の抑制効果が弱い

### 決定2: 超過の計算は `requiredCount > 0` のペアのみを対象とする

**採用**: `requiredCount <= 0` のスロットは評価対象外

**理由**: 必要人数が0のスロットへのアサインは「全時間帯出勤の原則」の観点から既に `getRequiredCount <= 0` で除外されており、超過計算でも同様の扱いとする。評価の一貫性を保つ。

### 決定3: `excessTotal` は合計値（総和）とする（ピークではなく）

**採用**: `Σ max(0, assigned - required)` over all (date, slot) pairs where `required > 0`

**理由**: 複数の小さな超過の合計を評価することで、特定の日時に超過が集中するより全体的に分散した超過を持つ解と等価に扱える。不足基準（`shortfallPeak` と `shortfallTotal`）と対称的に、超過も「合計」で評価する。

## Risks / Trade-offs

- **既存テストの更新が必要**: `EvalResult` を直接参照するすべてのテストに `excessTotal` フィールドを追加する必要がある。工数は大きくないが、漏れが TypeScript コンパイルエラーで検出される。

- **超過がゼロにならない場合がある**: 「全時間帯出勤の原則」により、不足を解消するためにスタッフを出勤させた際に別スロットで超過が発生することがある。この場合、`shortfallPeak/shortfallTotal` の改善を優先するため `excessTotal` が増加する解を採用することがある（設計上の意図した許容）。

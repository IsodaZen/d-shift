## Context

現在の自動シフト生成（`autoShiftGenerator.ts`）は、日付を先頭から順に処理するグリーディアルゴリズムで実装されている。各日付で週カウントの少ないスタッフを優先的にアサインするが、全体最適の保証はない。スタッフ10人未満・期間35日以下の小規模問題に特化した最適化が求められている。

既存の `generateAutoShift()` は純粋関数として実装されており、その出力 `ShiftAssignment[]` を最適化エンジンの入力（初期解）として利用する。

## Goals / Non-Goals

**Goals:**
- グリーディ初期解を局所探索で改善し、3段階の辞書式評価基準でより良いシフトを生成する
- 処理中にUIをブロックしない（Web Worker による非同期実行）
- 既存の強制制約をすべて維持する
- 10秒以内に最適化を完了する

**Non-Goals:**
- 厳密な最適解の保証（局所探索はヒューリスティック）
- 評価基準の重みやパラメータのUI設定
- 既存のグリーディアルゴリズム自体の変更
- 大規模問題（50人以上等）への対応

## Decisions

### 1. 最適化アルゴリズム: Hill Climbing（山登り法）

**選択**: 改善が見つかれば即座に採用する単純な山登り法
**理由**: 規模が小さい（<350セル）ため、シンプルなアルゴリズムでも十分な反復回数を確保できる。シミュレーテッドアニーリングやタブーサーチは実装複雑度に対する効果が見合わない。
**代替案**: シミュレーテッドアニーリング（温度パラメータの調整が必要）、遺伝的アルゴリズム（オーバーキル）

### 2. 内部表現: 二次元ビットマップ `working[staffIndex][dateIndex]`

**選択**: 最適化中は `boolean[][]` で出勤状態を管理し、最終的に `ShiftAssignment[]` に変換する
**理由**: 近傍操作と評価関数の高速化のため。`ShiftAssignment[]` は配列操作が多く反復に不向き。
**代替案**: `ShiftAssignment[]` を直接操作（各反復で配列フィルタリングが必要で遅い）

### 3. 評価関数: 辞書式比較（Lexicographic Comparison）

**選択**: 3基準を優先順位順に比較。第1基準が同値なら第2基準で比較、それも同値なら第3基準で比較
**理由**: ユーザーが明確な優先順位を指定しているため、重み付き線形結合より意図に忠実
**代替案**: 重み付き和（重みの調整が困難で直感に反する挙動の可能性）

### 4. 近傍操作

4種類の近傍操作をランダムに選択して適用する:

| 操作 | 内容 | 効果 |
|------|------|------|
| Toggle ON | 休み → 出勤 | 不足削減 |
| Toggle OFF | 出勤 → 休み | 超過削減・公平性改善 |
| Swap | 同日のスタッフ入替 | 公平性改善（不足数不変） |
| Move | 出勤日を別日に移動 | 不足の平準化 |

各操作は適用前に強制制約をチェックし、違反する場合は棄却する。

### 5. Web Worker 統合

**選択**: Vite の `new Worker(new URL(...), { type: 'module' })` パターン
**理由**: Vite でネイティブサポートされており、`?worker` import より明示的でデバッグしやすい
**通信**: `postMessage` で入力データ送信、プログレス通知、結果返却

### 6. 駐車場の扱い

**選択**: 最適化中は「各日の駐車場利用スタッフ数」のみを追跡し、具体的な駐車場スポット割り当ては最終結果の変換時に行う
**理由**: 近傍操作のたびにスポット割り当てを再計算するのは無駄。利用数の最大値さえ追跡すれば評価基準3を計算できる

### 7. ヘルプスタッフの扱い

**選択**: 最適化のスタッフ配列に通常スタッフとヘルプスタッフを統合し、ヘルプスタッフにはフラグで区別する
**理由**: 近傍操作と評価関数を統一的に扱える。ヘルプスタッフ固有の制約（availableDates、週上限なし）はフラグで分岐する

## Risks / Trade-offs

- **局所最適に陥るリスク** → ランダム要素を含む近傍選択と、十分な反復回数（~20,000回）で軽減。規模が小さいため局所最適でも実用上十分な品質が期待できる
- **Web Worker のデータ転送コスト** → 入出力データは小さい（<10KB）ため無視できる
- **ブラウザ互換性** → Web Worker と ES Modules は主要ブラウザで広くサポート済み
- **初期解の品質依存** → グリーディの初期解が極端に悪い場合は改善幅が限定的。ただし現実のデータでは初期解は概ね良好

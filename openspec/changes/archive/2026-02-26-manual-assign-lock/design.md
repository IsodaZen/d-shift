## Context

現在の `ShiftAssignment` 型には手動/自動の区別がなく、自動生成（bulkSetAssignments）はシフト作成期間内の全アサインを削除して上書きする。そのため、自動生成後に手動で行った調整が再生成で失われてしまう。

本設計では、アサインに「固定」フラグを導入し、固定アサインを自動生成から保護する。

## Goals / Non-Goals

**Goals:**
- 手動登録したアサインを固定アサインとして区別できる
- 自動生成が固定アサインを上書き・削除しない
- シフト表の各セルで固定/非固定をトグルできる（自動生成アサインを固定化、または固定を解除）
- シフト表上で固定アサインを視覚的に識別できる
- 既存の LocalStorage データとの後方互換性を保つ

**Non-Goals:**
- 固定アサインの一括解除機能
- 時間帯単位（時間帯ごと）の個別ロック切り替え（トグルはセル=スタッフ×日付 単位で動作する）

## Decisions

### 1. `ShiftAssignment` に `isLocked: boolean` フィールドを追加する

**Decision**: 別エンティティ（`AssignLock`）を作るのではなく、既存の `ShiftAssignment` 型に `isLocked` フィールドを直接追加する。

**Rationale**: ロック状態はアサインと1対1で対応し、常にアサインと一緒に読み書きされる。別テーブルを設けると JOIN 相当の処理が必要になり複雑度が増す。また LocalStorage の読み書きも1回で済む。

**Alternative**: `AssignLock { staffId, date }` を別 localStorage キーで管理 → アサインと別々に管理するためデータの整合性維持が難しくなる。

---

### 2. ロックの判定単位はスタッフ + 出勤日（staff + date）

**Decision**: 特定の (staffId, date) ペアに `isLocked: true` のアサインが1つでも存在すれば、そのスタッフのその日を「固定日」とみなす。

**Rationale**: 要件「固定はスタッフと出勤日単位に設定する」に対応。手動登録は「その日の全シフト構成」を確定させる操作として扱い、自動生成でその日のスタッフ割り当てを変更しない。

---

### 3. 自動生成アルゴリズムは固定日を入力として受け取りスキップする

**Decision**: `generateAutoShift` に `lockedStaffDates: Set<string>` (key: `"staffId_date"`) を渡し、その組み合わせのスタッフ・日付への生成を完全にスキップする。固定日のアサインは自動生成の結果に含めない（呼び出し元でマージする）。

**Rationale**: 生成アルゴリズムを純関数のまま保ちつつ、固定制約を外部から注入できる。

**Alternative**: アルゴリズム内部で existing assignments を参照してスキップ → 関心の混在でテストが複雑になる。

---

### 4. bulkSetAssignments は固定アサインを保護する

**Decision**: `bulkSetAssignments` を変更し、`periodDates` 内のアサインを削除する際に `isLocked: true` のアサインを除外（保持）する。

**Rationale**: 自動生成側のアルゴリズムが固定日をスキップするため、生成結果と既存の固定アサインは同じ (staffId, date) を重複して含まない。両者を合わせることで整合性を保てる。

---

### 5. LocalStorage の既存データ移行は読み込み時のデフォルト適用で対処

**Decision**: `useLocalStorage` の初期値では対応できないため、`useAssignments` の内部で旧データ読み込み時に `isLocked` が `undefined` の場合は `false` にフォールバックする処理を追加する。

**Rationale**: スキーマバージョン管理は複雑すぎる。フィールドのデフォルト値で十分。

### 6. セルの固定トグルはスタッフ × 日付 単位で全アサインを一括更新する

**Decision**: トグルボタンはセル（staffId + date）単位で動作し、そのセルに属する全 `ShiftAssignment` の `isLocked` を一括で反転する。

- セルに `isLocked: false` のアサインが1件でも存在する場合 → 全アサインを `isLocked: true` に更新（固定化）
- セルの全アサインが `isLocked: true` の場合 → 全アサインを `isLocked: false` に更新（固定解除）
- アサインが存在しないセルにはトグルボタンを表示しない

**Rationale**: 要件「固定はスタッフと出勤日単位に設定する」に対応。時間帯ごとの部分ロックを許容すると、ロック状態の判定が複雑になり、「固定日はスキップ」という自動生成ロジックとの整合性が崩れる。

**Alternative**: 時間帯単位でのトグル → 部分ロック（午前は固定・午後は自動生成対象）が発生し、"全時間帯アサイン制約" との相互作用が複雑になるため採用しない。

---

## Risks / Trade-offs

- **[リスク] 手動アサイン後の自動生成で固定分のシフト枠が充足扱いになる** → 固定アサインのスロットは `remaining` カウントを減算し、残り枠のみを自動生成対象にすることで対応。ただし固定アサインのスタッフ全体をスキップするため、固定日のスロット構成は変わらない。
- **[リスク] 手動でアサインを削除した後も「固定」が残る** → アサイン削除（removeAssignment）で当該アサインが消えると、同日の他アサインが isLocked: true であれば固定は維持される。全アサイン削除後は固定解除と同等になる。特別な処理不要。
- **[トレードオフ] 手動登録 = 常に固定** → 手動登録直後は常に `isLocked: true` になる。固定にしたくない場合はトグルで解除できるため許容。
- **[トレードオフ] セル単位のトグルで部分ロックが一時的に発生する** → 「午前は固定・午後は非固定」の状態を経由して最終的に全固定にするユースケースがあるが、自動生成はその時点で固定日判定をするため、中間状態で自動生成を走らせると午後が再生成される。ユーザーが意図する操作完了（トグル後）に自動生成すれば問題ない。

## Migration Plan

1. `src/types/index.ts` の `ShiftAssignment` に `isLocked: boolean` を追加
2. `useAssignments` の読み込み時に旧データの `isLocked` 不在を `false` にフォールバック（型アサーションで吸収）
3. `addAssignment` を `isLocked: true` で保存するよう変更
4. `bulkSetAssignments` を固定アサイン保護ロジックへ変更
5. `generateAutoShift` に `lockedStaffDates` パラメータを追加
6. `useAssignments` に `setCellLocked(staffId, date, isLocked)` を追加
7. シフト表のセルコンポーネントに固定トグルボタンを追加

**Rollback**: LocalStorage を削除するか、アプリをリロードするだけで旧データに戻せる。データ移行は読み込み時の互換処理のみのため、ロールバックコードは不要。

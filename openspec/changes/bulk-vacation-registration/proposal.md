## Why

現在の希望休登録UIは、スタッフと日付を1つずつ選択して「登録」ボタンを押す方式（1回1件）であり、以下の問題がある。

1. **登録作業が非効率**: 複数日の希望休をまとめて登録したい場合、スタッフを選択し直すことなく繰り返しの操作が必要になる。例えば、あるスタッフが1か月分のシフト対象期間に8日の希望休を持つ場合、8回の操作が必要になる。
2. **入力ミスのリスク**: 1件ずつ登録する方式では、スタッフを間違えたまま複数件登録するミスが起きやすい。
3. **UXの断絶**: 実際の運用では「このスタッフは○日、△日、□日が休み」とまとめて把握されており、UIがその実態と合っていない。
4. **削除操作が煩雑**: 個別エントリを探して削除ボタンを押す必要があり、複数件の修正が難しい。

スタッフを選択した後、カレンダーUIで希望休を一覧把握しながら複数日付を編集・保存できる機能に刷新することで、登録・修正・削除のすべてを一画面で完結させる。

## What Changes

### 希望休の編集UIをカレンダー型に刷新

- **スタッフ選択**: 既存のセレクトボックスを維持する
- **カレンダー表示**: スタッフ選択後、シフト期間を含む月のカレンダーを表示する
  - すでに登録済みの日付は最初から選択済み（ハイライト）状態で表示する
  - 日付をタップするとトグル（選択 ↔ 解除）できる
  - シフト期間外の日付はグレーアウトして選択不可とする
  - シフト期間が設定されていない場合: 既存の単一日付入力（`<input type="date">`）を維持する（フォールバック）
- **保存（同期）**: 「保存」ボタンを押すと、カレンダーの選択状態を正として希望休をLocalStorageに同期する
  - 新たに選択された日付 → 追加登録
  - 選択を解除された登録済み日付 → 削除
  - 変化のない日付 → そのまま

### 希望休一覧の表示をスタッフ別サマリーに変更

- **変更前**: スタッフ名・日付のリスト（個別エントリ＋削除ボタン）
- **変更後**: スタッフごとの行に「シフト期間内の希望休件数」を表示する
  - 希望休が1件以上登録されているスタッフのみ表示する
  - カレンダーUIで編集・削除するため、個別削除ボタンは不要

### `useDayOffs` フックへの `syncDayOffs` 追加

現在の `addDayOff`（単数形）は1件ずつ登録する。カレンダーUIの「現在の選択状態」をLocalStorageに同期する `syncDayOffs` 関数を追加する。

```ts
// 追加するシグネチャ（イメージ）
syncDayOffs(staffId: string, dates: string[]): { added: number; removed: number }
```

- `added`: 新規登録された件数
- `removed`: 削除された件数

## Capabilities

### Modified Capabilities

- `preferred-day-off`: 希望休登録UIをカレンダーUIによる編集・保存（同期）方式に変更する。登録済み日付をカレンダー初期選択状態で表示し、選択解除＋保存で削除できるようにする。希望休一覧をスタッフ別の件数サマリーに変更する。`useDayOffs` フックに `syncDayOffs` を追加する

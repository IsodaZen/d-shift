## Why

現在の希望休登録UIは、スタッフと日付を1つずつ選択して「登録」ボタンを押す方式（1回1件）であり、以下の問題がある。

1. **登録作業が非効率**: 複数日の希望休をまとめて登録したい場合、スタッフを選択し直すことなく繰り返しの操作が必要になる。例えば、あるスタッフが1か月分のシフト対象期間に8日の希望休を持つ場合、8回の操作が必要になる。
2. **入力ミスのリスク**: 1件ずつ登録する方式では、スタッフを間違えたまま複数件登録するミスが起きやすい。
3. **UXの断絶**: 実際の運用では「このスタッフは○日、△日、□日が休み」とまとめて把握されており、UIがその実態と合っていない。

スタッフを選択した後、複数の日付をまとめて選択し一括登録できる機能を追加することで、登録作業の効率を向上させる。

## What Changes

### 希望休の一括登録UI

- **スタッフ選択**: 既存のセレクトボックスを維持する
- **日付の複数選択**: 日付を複数選択できるUIを提供する
  - シフト期間が設定されている場合: 期間内の日付一覧をチェックボックス形式で表示する
  - シフト期間が設定されていない場合: 既存の単一日付入力（`<input type="date">`）を維持する（フォールバック）
- **一括登録**: 選択されたスタッフと複数日付をまとめて `addDayOff` を呼び出し登録する
  - 既存データと重複する日付はスキップし、新規分のみ登録する
  - 登録完了後、選択済み日付をクリアする

### `useDayOffs` フックへの `addDayOffs`（複数形）の追加

現在の `addDayOff`（単数形）は1件ずつ登録する。複数件を一度に登録する `addDayOffs` 関数を追加する。

```ts
// 追加するシグネチャ（イメージ）
addDayOffs(staffId: string, dates: string[]): { added: number; skipped: number }
```

- `added`: 新規登録された件数
- `skipped`: 重複のためスキップされた件数

## Capabilities

### Modified Capabilities

- `preferred-day-off`: 希望休登録UIに複数日付選択（チェックボックス一覧）を追加し、一括登録に対応する。シフト期間未設定時は従来の単一日付入力にフォールバックする。`useDayOffs` フックに `addDayOffs` を追加する
